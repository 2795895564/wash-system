# 后端接口对接文档-系统设置（后端实现版）

> 面向：后端开发 / AI 生成后端代码
>
> 目标：一次性把“系统设置”模块的接口补齐并与前端联调通过。
>
> 覆盖模块：
> - 字典管理（sys_dict / sys_dict_item）
> - 参数配置（sys_config）
> - 操作日志（sys_oper_log）

---

## 0. 为什么要出“后端实现版”

当前仓库存在两类事实：

- **前端已按项目 CURD 规范编写页面，并封装了部分接口调用**（例如 `GET /{id}/form` 用于编辑回显）。
- **原对接文档中缺失部分接口**，导致后端未实现时，前端会直接请求失败。

本文件用于：

- 明确哪些接口是**前端正在调用**且**后端必须实现**
- 给出 **路径 / 入参 / 出参 / 错误码 / 实现建议**

---

## 1. 通用约定

### 1.1 Base URL
- 后台接口统一前缀：`/api/admin`

### 1.2 鉴权
- 请求头携带：`Authorization: Bearer <token>`

### 1.3 通用响应结构（AdminResult）
所有接口统一返回：
```json
{
  "code": "00000",
  "data": {},
  "msg": "操作成功"
}
```

### 1.4 分页结构（与现有接口保持一致）
分页接口的 `data` 推荐结构：
```json
{
  "total": 0,
  "records": [],
  "pages": 0
}
```

### 1.5 逻辑删除（deleted）
- 使用 MyBatis-Plus 逻辑删除字段：`deleted`
- `deleted=0` 未删除
- `deleted=1` 已删除

### 1.6 错误码（最低要求）
- `A0300`：参数错误/资源不存在/不允许的操作

## 2. 字典管理（sys_dict / sys_dict_item）

### 2.0 字段约定（后端返回字段）
字典（sys_dict）建议字段：
- `id`
- `dictCode`
- `dictName`
- `status`（1/0）
- `remark`
- `createTime` / `updateTime`

字典项（sys_dict_item）建议字段：
- `id`
- `dictCode`
- `itemValue`
- `itemLabel`
- `tagType`
- `sort`
- `status`
- `remark`
- `createTime` / `updateTime`

> 备注：前端存在 tagType 的编码/解码逻辑，后端可选择两种返回方案：
> - 方案A：直接返回 `primary/success/warning/info/danger/""`
> - 方案B：返回单字符编码 `P/S/W/I/D/N`（若返回编码，需与前端约定一致）

---

### 2.1 字典分页列表（已存在）
#### GET `/api/admin/system/dict/page`

Query：
- `page`：默认 1
- `size`：默认 10
- `keyword`：可选（匹配 dictCode/dictName）
- `status`：可选（1启用/0禁用）

Response.data：
```json
{
  "total": 1,
  "records": [
    {
      "id": 1,
      "dictCode": "gender",
      "dictName": "性别",
      "status": 1,
      "remark": "系统内置字典",
      "createTime": "2026-03-16 10:00:00",
      "updateTime": "2026-03-16 10:00:00",
      "deleted": 0
    }
  ],
  "pages": 1
}
```

---

说明：

### 2.5 【必须实现】字典表单回显（前端编辑依赖）
#### GET `/api/admin/system/dict/{id}/form`

用途：
- 前端点击“编辑字典”时调用该接口拉取表单数据。

Response.data：
```json
{
  "id": 1,
  "dictCode": "gender",
  "dictName": "性别",
  "status": 1,
  "remark": "系统内置字典"
}
```

错误：
- 不存在：`code=A0300`，`msg=字典不存在`

实现建议：
- `SELECT ... FROM sys_dict WHERE id=? AND deleted=0`

## 4. 操作日志（sys_oper_log）

### 4.1 操作日志分页查询（已存在）
#### GET `/api/admin/system/log/page`

Query：
- `page`：默认 1
- `size`：默认 10
- `startDate`：可选（YYYY-MM-DD）
- `endDate`：可选（YYYY-MM-DD）
- `userId`：可选
- `success`：可选（1成功/0失败）
- `keyword`：可选（匹配 path/username）

Response.data：
```json
{
  "total": 10,
  "records": [
    {
      "id": 1,
      "userId": 4,
      "username": "管理员",
      "method": "GET",
      "path": "/api/admin/system/dict/page",
      "ip": "127.0.0.1",
      "userAgent": "PostmanRuntime/7.x",
      "requestParams": "page=1&size=10",
      "responseCode": "200",
      "success": 1,
      "costMs": 12,
      "createTime": "2026-03-16 17:10:00",
      "deleted": 0
    }
  ],
  "pages": 1
}
```

---

### 4.2 （可选）日志详情
当前前端“查看详情”使用列表行数据即可，但如果后端需要返回更长字段（例如 requestParams 很长），可补接口：

#### GET `/api/admin/system/log/{id}`
Response.data：返回单条完整日志对象。

## 5. 后端实现优先级（让 AI/后端按顺序做）

### 5.1 必须实现（否则前端直接报错）
- `GET /api/admin/system/dict/{id}/form`

### 5.2 强烈建议实现（保证编辑体验与数据一致）
- `GET /api/admin/system/dict/{dictCode}/items/{id}/form`
- `GET /api/admin/system/config/{id}/form`

### 5.3 可选增强
- `GET /api/admin/system/dict/options`
- `POST /api/admin/system/config/refresh-cache`
- `GET /api/admin/system/log/{id}`
